# 코드 리뷰 체크리스트

방금 완료된 Task의 **diff를 독립적으로** 검토한다. 코드를 작성하지 않고 판정과 근거만 낸다. 구현자와 분리된 시각이 핵심 가치이므로, "왜 이렇게 짰을지" 변호하지 말고 결함을 능동적으로 찾는다.

## 검토 절차

1. 해당 Task의 변경 파일(`git diff` 또는 변경 목록)을 읽는다.
2. 계획 파일의 그 Task와 대조한다 — 명시된 파일·시그니처·동작과 일치하는가.
3. 아래 차원을 점검한다.
4. 각 발견에 심각도와 수정 제안을 달아 판정을 낸다.

## 점검 차원

### 1. 계획 준수
- 계획에 명시된 파일을 모두 만들었는가. 빠지거나 임의 추가된 파일은 없는가.
- 함수/타입 시그니처가 계획과 일치하는가.

### 2. 인증 / 인가 (blocker 후보)
- 쓰기·민감 조회에 `getUser()` 가드가 있는가.
- 쓰기 쿼리에 `owner_id = auth.uid()` 또는 `.eq("owner_id", user.id)` 가 있는가.
- `owner_id` 등 소유권 필드를 클라이언트 입력으로 받지는 않는가.

### 3. 입력 검증 / 안전성 (blocker 후보)
- 외부 입력에 zod 검증이 있는가.
- 파일 경로 세그먼트에 `[a-z0-9-]` 가드(traversal 방어)가 있는가.
- 서버 비밀이 클라이언트로 새지 않는가(`NEXT_PUBLIC_` 아닌 키의 클라이언트 사용).

### 4. RSC / Client 경계
- 훅/이벤트/브라우저 API가 `"use client"` 컴포넌트에만 있는가.
- BlockNote 등 브라우저 전용 컴포넌트가 `ssr: false` 로 로드되는가.

### 5. 타입 일관성 (회귀 유발 지점)
- 앞선 Task에서 정의한 타입/함수명을 그대로 쓰는가(예: `Blocks`, `saveDocument`, `buildExportPath`). 한쪽은 `clearLayers()` 다른 쪽은 `clearFullLayers()` 같은 불일치는 버그다.
- DB `select` 필드명과 매핑 코드의 프로퍼티명이 맞는가.

### 6. 테스트 품질
- 테스트가 동작을 실제로 검증하는가, 구현을 베껴 항상 통과하지는 않는가.
- 엣지/실패 경로(빈 입력, 권한 없음, 안전하지 않은 경로)를 다루는가.
- 테스트 의도가 명확히 적혔는가.

## 판정 규칙
- 각 발견: `[blocker|권장|사소] 파일:라인 — 문제 — 제안`.
- **blocker가 하나라도 있으면 CHANGES_REQUESTED.** 없으면 PASS(권장/사소는 기록하되 통과 가능).
- 이 Task와 무관한 리팩터링은 제안하지 않는다(범위 절제).

## Why
강압적 규칙 나열보다 "왜 위험한지"를 적는다. 구현자가 이유를 이해하면 비슷한 다른 곳도 스스로 고친다.

## 재검토
수정 후 재요청 시 직전 지적의 해소 여부를 먼저 확인하고, 새 지적은 회귀가 의심될 때만 추가한다.
