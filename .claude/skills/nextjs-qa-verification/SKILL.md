# Next.js QA 검증

구현을 **실제로 실행**해 동작과 경계면 정합성을 확인한다. "파일이 있다"가 아니라 "동작이 맞고 양쪽이 맞물린다"를 증명하는 것이 목적이다. 결함은 일찍 잡을수록 싸므로, 전체 완성 후 한 번이 아니라 **각 Task/모듈 직후 점진적으로** 돈다.

## 실행 검증

1. **타입:** `pnpm tsc` — 타입 에러 0.
2. **단위/회귀:** `pnpm test` — **전체 스위트**를 돈다. 신규 테스트만 보지 않는다. 이전 기능이 깨졌는지 확인하기 위함이다.
3. **E2E — 슬라이스 완료 시 1회**(매 Task마다 X, 무겁다): `pnpm e2e`. 단위 테스트(vitest)는 **CSR만** 하므로 **hydration mismatch·런타임 에러를 구조적으로 못 잡는다** — e2e가 실제 브라우저로 페이지를 로드해 **콘솔 error/warning + pageerror 0건**을 단언하는 것이 그 갭을 메운다. 세션 주입(`storageState`)으로 인증 후 페이지(`/`, `/edit`, `/p`)까지 본다. 테스트 유저 env(`E2E_TEST_EMAIL`/`E2E_TEST_PASSWORD`) 없으면 인증 스모크는 **미검증으로 분리**(anon 스모크만 실행).

> tsc·단위/회귀는 매 Task/모듈 직후 점진적으로, **e2e는 슬라이스 전체 완료 시점에만** 돈다.

출력을 근거로 판정한다. 읽기만으로 통과를 적지 않는다.

### 회귀 처리
- 기존 테스트가 깨지면 어떤 테스트가 왜 깨졌는지 명시한다.
- **기존 테스트를 고치라고 제안하지 않는다** — 신규 코드를 고쳐야 한다(CLAUDE.md §3). 단, 의도된 동작 변경으로 보이면 그 사실을 보고한다.

## 경계면 교차 비교 (QA의 본질)

버그는 한 파일 안이 아니라 **두 모듈이 만나는 지점**에 숨는다. 한쪽만 보면 못 잡는다. 양쪽을 함께 읽어 shape을 대조한다.

대조할 대표 경계면:
- **API 반환 ↔ 클라이언트 소비.** 예: export 라우트가 `{ ok, path }` 를 반환하는데, `editor.tsx` 가 `body.path` 를 읽는가. 키 이름·존재·타입이 맞는가.
- **DB 컬럼/`select` ↔ 매핑 코드.** `select("owner_id, created_at")` 의 snake_case 필드를 `get.ts`/`list.ts` 가 올바른 키로 읽어 camelCase로 매핑하는가.
- **zod 스키마 ↔ 실제 데이터.** 액션이 기대하는 형태로 클라이언트가 보내는가.
- **서버 액션 시그니처 ↔ 호출부 인자 순서/타입.**

불일치는 `[심각도] 위치 — 한쪽 형태 vs 다른쪽 형태 — 영향` 으로 보고한다.

## 환경 의존 구분

외부 자원이 없으면 검증 가능/불가능을 분리한다. 못 한 검증을 통과로 적지 않는다.
- Supabase 연결 없음 → DB 왕복 동작은 미검증으로 표시.
- `WIKI_ROOT` 미설정 → export 파일 쓰기는 미검증(라우트가 503 반환하는지만 확인 가능).
- 에러가 환경 문제(미설치·키 부재)인지 코드 결함인지 구분해 보고한다.

## 보고 형식
- 실행 명령 + 결과(통과/실패 수, 실패 시 핵심 출력)
- 경계면 비교 결과(대조 양쪽 + 일치/불일치)
- 결함: `[심각도] 위치 — 증상 — 재현/근거`
- 미검증 항목과 이유

## 재검증
결함 수정 후에는 이전에 실패한 케이스를 먼저 재실행해 해소를 확인하고, 전체 스위트로 회귀를 본다.
