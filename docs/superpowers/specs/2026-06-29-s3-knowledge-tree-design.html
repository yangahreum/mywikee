S3 — 지식 트리 (사이드바 Knowledge Tree)

사이드바 왼쪽 메뉴의 KNOWLEDGE TREE 영역에 폴더 계층 트리를 구현한다(청사진 Frame 1). 폴더와 문서를 최대 4단 깊이로 계층화하고, 항목을 클릭해 열고(폴더 토글 / 문서 에디터 이동) 드래그앤드롭으로 재배치한다.

folders 신규 테이블4단 깊이클릭 열기 + DnDSupabase RLSvitest

1. 배경 & 범위

현재 documents는 평면(project 텍스트 필드뿐)이다. S3는 폴더 계층을 도입해 사이드바 Knowledge Tree(현 placeholder)를 실데이터로 채운다.

범위 (S3). ① folders 테이블 + documents.folder_id 데이터 모델, ② 트리 조회/빌드, ③ 사이드바 트리 UI(청사진 Frame 1 픽셀, 펼침/접힘), ④ 폴더 생성(plus-square), ⑤ 클릭 열기(폴더 토글 / 문서→/edit), ⑥ 드래그앤드롭 재배치(문서·폴더를 다른 폴더로, 4단 제약 검증), ⑦ 사이드바 통일(아래 §5.0).

1.1 사이드바 통일 (결정 2026-06-29)

청사진은 사이드바가 화면마다 달랐다 — 홈(Frame 1)은 Home·Recent + KNOWLEDGE TREE, 에디터/아티클(Frame 2/3)은 flat 네비(Knowledge Base·Recent Notes·Favorites·Folders·Archive). 사용자 결정: 왼쪽 사이드바는 모든 화면 공통으로 트리를 고정한다. 따라서 청사진 Frame 1 사이드바(Home·Recent + KNOWLEDGE TREE)를 공통 사이드바로 채택하고, Frame 2/3의 flat 네비는 쓰지 않는다. (아티클의 "오른편 네비게이션" = TOC/Related는 읽기뷰 S5 소관.)

  • S1에서 만든 NAV_ITEMS(Home/Recent/Knowledge Base/Favorites/Folders/Archive 6개)를 Home·Recent 2개로 축소하고, 그 아래에 KNOWLEDGE TREE를 둔다. Knowledge Base/Favorites/Folders/Archive 네비 항목은 트리로 대체(해당 화면은 후속).
  • nav.ts의 NAV_ITEMS와 그 테스트를 이 결정에 맞춰 갱신(요구사항 변경에 따른 스펙 동기화 — 신규 기능 충돌이 아니라 의도된 변경).
  • 사이드바 너비는 공통 228px(현재 에디터 212px도 228px로 통일). 트리가 들어가므로 홈 기준 폭 사용.
비범위 (후속). 폴더 이름변경·삭제(S3 확장), Favorites/Archive/Recent 화면(별도 슬라이스 — 네비 항목은 이미 있으나 화면은 후속), 폴더별 문서 목록 전용 페이지(트리에서 펼치는 것으로 갈음), 다중 선택·키보드 DnD. documents.project 필드는 당분간 유지(폴더와 병존, 추후 정리).

2. 데이터 모델

2.1 folders 테이블 (신규)

CREATE TABLE IF NOT EXISTS public.folders (
  id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_id   uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  parent_id  uuid REFERENCES public.folders(id) ON DELETE CASCADE,  -- NULL = 루트
  name       text NOT NULL DEFAULT '새 폴더',
  depth      int  NOT NULL DEFAULT 1 CHECK (depth BETWEEN 1 AND 4), -- 1=루트 … 4=최대
  position   int  NOT NULL DEFAULT 0,                                -- 같은 parent 내 정렬
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
-- updated_at 트리거: 기존 set_updated_at() 재사용
CREATE INDEX IF NOT EXISTS idx_folders_owner_parent ON public.folders (owner_id, parent_id, position);
  • depth를 컬럼으로 캐시해 4단 제약을 DB(CHECK)와 앱 양쪽에서 강제. 루트 폴더 depth=1.
  • ON DELETE CASCADE: 부모 폴더 삭제 시 하위 폴더 연쇄 삭제(삭제는 후속이나 FK는 지금 설정).

2.2 documents 변경

ALTER TABLE public.documents
  ADD COLUMN IF NOT EXISTS folder_id uuid REFERENCES public.folders(id) ON DELETE SET NULL;
-- 문서는 폴더 안(folder_id) 또는 루트(NULL). 폴더 삭제 시 문서는 루트로(SET NULL).
CREATE INDEX IF NOT EXISTS idx_documents_owner_folder ON public.documents (owner_id, folder_id);

2.3 RLS

ALTER TABLE public.folders ENABLE ROW LEVEL SECURITY;
DROP POLICY IF EXISTS "owners full access on own folders" ON public.folders;
CREATE POLICY "owners full access on own folders"
  ON public.folders FOR ALL
  USING (auth.uid() = owner_id) WITH CHECK (auth.uid() = owner_id);
  • 마이그레이션: supabase/migrations/0003_folders.sql (또는 publish가 0003이면 다음 번호). youllog 수동 적용: docs/setup/folders.sql(재실행 안전).

3. 4단 깊이 제약

  • 폴더 생성: 부모 depth + 1 ≤ 4. 부모가 depth 4면 하위 폴더 생성 불가(UI에서 plus-square 비활성/토스트).
  • 폴더 이동(DnD): 이동 후 서브트리 전체의 최대 depth ≤ 4 여야 한다. 즉 (대상 부모 depth) + (이동 폴더의 서브트리 높이) ≤ 4. 위반 시 드롭 거부(시각 피드백). 이동 시 서브트리 depth 일괄 갱신.
  • 문서는 depth를 차지하지 않음(폴더 안의 잎). 문서는 depth 4 폴더에도 들어갈 수 있다.
  • 순환 방지: 폴더를 자기 자신 또는 자기 자손으로 이동 금지(서버 검증).
  • depth 계산·검증은 순수 함수로 분리(lib/tree/depth.ts) → TDD.

4. 트리 조회 & 빌드

  • lib/tree/get-tree.ts: owner의 folders 전체 + documents(id,title,folder_id) 조회 → 메모리에서 계층 트리 빌드.
  • 빌드 순수 함수 buildTree(folders, docs) → TreeNode[](폴더 노드는 children + 문서, position 정렬) — TDD.
  • 타입: TreeFolder{id,name,parentId,depth,position}, TreeDoc{id,title,folderId}, TreeNode = {kind:'folder', ...children} | {kind:'doc', ...}.

  • 폴더 행(h32, text-[13px]): chevron-down/chevron-right(15) + folder(15) + 이름. 펼침 폴더 이름 text-ink/500, 접힘 text-ink-secondary.

6. 인터랙션

6.1 클릭 열기

  • 폴더 클릭: 펼침/접힘 토글(expanded 클라이언트 상태). chevron 회전/교체.
  • 문서 클릭: /edit/{id}로 이동(Link).
  • 접힘/펼침 상태는 클라이언트 상태(로컬). 영속화는 후속(localStorage 등).

6.2 드래그앤드롭 재배치

  • 라이브러리: @dnd-kit/core(+ @dnd-kit/sortable 필요 시) — React 19 호환, 경량, 접근성. (대안 react-arborist는 트리 특화지만 무겁고 커스텀 제약 → dnd-kit + 자체 트리 권장. 구현 계획에서 최종 확정.)
  • 드래그 대상: 폴더, 문서. 드롭 대상: 폴더(그 안으로), 루트 영역.
  • 드롭 시 서버 액션 moveItem 호출 → 낙관적 UI 갱신 + 실패 시 롤백.
  • 드롭 가능 여부 시각화: 유효(하이라이트) / 무효(4단 초과·순환 → 비활성 표시).

7. 서버 액션

파일: app/tree/actions.ts (또는 lib/tree/actions.ts, 구현 계획서에서 확정). 모두 getUser 가드 + owner 이중조건 + zod.

  • createFolder(parentId: uuid|null, name?): 부모 depth+1 ≤ 4 검증 → insert(depth, position=말미). 반환 새 폴더.
  • moveItem({kind:'folder'|'doc', id, targetParentId: uuid|null}):
    • doc: documents.folder_id = targetParentId 업데이트.
    • folder: 순환 검사 + (target depth + 서브트리 높이) ≤ 4 검증 → parent_id 변경 + 서브트리 depth 일괄 재계산.
  • 오류: unauthorized/validation/depth_exceeded/cycle/not_found/db_error. revalidatePath("/") 등.

9. 테스트

  • vitest 단위:
    • depth: 부모 depth+1 검증(4 경계), 서브트리 높이 계산, 순환 검사(자기/자손 이동 거부).
    • build-tree: 평면 folders+docs → 계층(정렬·문서 배치·루트 처리), 빈 입력.
    • KnowledgeTree/TreeNodeRow: 폴더/문서 렌더, 펼침/접힘, 들여쓰기 depth, plus-square(루트 depth4 자식 없음 — 여기선 루트라 항상 가능), 활성 문서 표시.
    • (DnD 동작은 단위로 부분만 — 드롭 유효성 판정 순수 함수 위주. 실제 드래그는 QA/수동.)
  • 회귀: 기존 + S1/S2 동반 통과. Sidebar 변경이 기존 Sidebar 테스트를 깨지 않게(트리 prop은 옵셔널/기본 빈 배열).
  • 런타임/QA: 폴더 생성 → 트리 표시 → 문서 드래그로 폴더 이동 → 새로고침 유지; 4단 초과 드롭 거부; 순환 이동 거부. 실제 Supabase.

10. 미해결 / 검토

  • DnD 라이브러리: @dnd-kit/core(권장) vs react-arborist — 구현 계획에서 확정(번들·React19 호환·커스텀 자유도).
  • AppShell tree prop 추가가 모든 셸 페이지에 트리 조회를 요구 — 공통 레이아웃(route group)으로 한 번만 조회하도록 리팩토링할지(후속) vs 페이지별 조회(지금).
  • documents.project ↔ folder_id 병존: S2 태그칩/브레드크럼이 아직 project 사용. 폴더 도입 후 정리 시점은 후속.
  • 펼침 상태 영속화(localStorage), 폴더 빈 상태("폴더 없음 — + 로 생성") 처리.
  • 문서 생성 시 현재 폴더 컨텍스트에 넣을지(New Entry가 선택 폴더에) — 기본 루트, 후속 개선.
