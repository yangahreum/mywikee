Publish 기능 (퍼블릭 공개) — 설계

문서를 발행(draft → published)하면 비로그인 사용자를 포함한 누구나 공개 URL(/p/[slug])로 읽을 수 있게 한다. "공유"를 켜는 첫 지점.

1. 배경 & 목표

현재 위키 앱은 동작하지만 모든 문서가 소유자 전용이다 — documents의 RLS가 auth.uid() = owner_id 하나뿐이라, Supabase를 쓰더라도 실제로는 "공유"가 아니다. 이번 슬라이스는 Publish(퍼블릭 공개) 기능을 추가한다: 작성자가 문서를 발행하면 익명 포함 누구나 공개 URL로 읽을 수 있다. 즉 Publish가 공유를 켜는 첫 지점이다.

범위. 이번 슬라이스는 Publish 기능만 다룬다. 기존 source 생성(LLM 위키 export)과는 별개이며 그대로 둔다. 앱 셸·디자인 시스템·지식 트리·읽기 뷰 디자인·검색은 별도 슬라이스(S1~S6). 멤버 전용 공개(로그인 사용자만)는 나중에 RLS 정책 교체만으로 전환 가능하도록 둔다.

2. 데이터 흐름

[작성자] 에디터에서 "Publish" 클릭 → 서버 액션 setDocumentPublished(id, true) [owner 가드] → documents.status='published', published_at=now() │ ▼ [익명 방문자] GET /p/{slug} → 익명 Supabase 조회 (RLS: status='published' 공개) → content(Block[]) → HTML(blocksToHtml) → sanitize → 읽기 전용 렌더

4. RLS 정책 (핵심)

기존 "owner 전체 접근" 정책은 유지하고, 공개 읽기 정책을 추가한다.

-- 기존(유지): owner 전체 접근
--   "owners full access on own documents" : ALL USING/ WITH CHECK (auth.uid() = owner_id)

-- 신규: published 문서는 누구나(익명 포함) SELECT
DROP POLICY IF EXISTS "published documents are readable by anyone" ON public.documents;
CREATE POLICY "published documents are readable by anyone"
  ON public.documents FOR SELECT
  USING (status = 'published');

두 SELECT 정책은 OR로 합쳐진다 — 작성자는 자기 draft+published 전부, 익명/타인은 published만 읽는다. 쓰기(INSERT/UPDATE/DELETE)는 여전히 owner만. 발행 전환은 owner가 UPDATE하므로 기존 정책으로 충분하다.

⚠️ 멤버 전용으로 바꿀 때. 나중에 "로그인 멤버만 공개"로 전환하려면 이 정책의 USING (status='published')를 USING (status='published' AND auth.role() = 'authenticated')로 바꾸기만 하면 된다. 라우트/액션 변경 불필요.

5. 서버 액션: 발행 토글

파일: app/edit/[id]/actions.ts에 추가(기존 saveDocument 옆).

"use server";
// setDocumentPublished(id, published)
//  ① getUser 가드 (미인증 → unauthorized)
//  ② zod: { id: uuid, published: boolean }
//  ③ update documents set status, published_at
//     - publish:  status='published', published_at = now()
//     - unpublish: status='draft'   (published_at 유지)
//     .eq("id", id).eq("owner_id", user.id)  // owner 이중 보호
//  ④ PGRST116 → not_found / 기타 → db_error
//  ⑤ revalidatePath("/me"), revalidatePath(`/p/${slug}`)
// 반환: { ok:true, status, slug } | { ok:false, error }

반환에 slug를 포함해 클라이언트가 공개 URL(/p/{slug})을 즉시 보여줄 수 있게 한다.

6. 에디터 UI (기능 우선)

파일: app/edit/[id]/editor.tsx. 디자인 고정 툴바 챙은 S4 몫이므로, 이번엔 동작 가능한 버튼만 추가한다.

  • 현재 상태에 따라 Publish(draft일 때) / Unpublish(published일 때) 토글 버튼.
  • published면 공개 URL 표시 + 복사 버튼({origin}/p/{slug}).
  • 기존 source 생성 버튼은 그대로 둔다(별개 기능).
  • 진입 시 현재 status를 알아야 하므로 getDocument/page.tsx가 status를 함께 로드해 Editor에 전달.

7. 공개 읽기 라우트 /p/[slug]

파일: app/p/[slug]/page.tsx (서버 컴포넌트, 인증 불필요).

  • 익명 Supabase 클라이언트로 documents에서 slug로 1건 조회(status='published'만 RLS로 통과).
  • 없으면 notFound()(draft도 익명에겐 안 보이므로 404).
  • content(Block[]) → blocksToHtml(기존 lib/export/html.ts 재사용)로 HTML 생성.
  • 공개 페이지이므로 XSS sanitize 후 렌더(아래 §8). 읽기 전용 — 편집 UI 없음.
  • 제목 + 본문만 최소 렌더(본격 읽기 뷰 디자인·TOC는 S5). 메타로 published_at 표시 가능.

조회 헬퍼

lib/documents/get-published.ts: getPublishedDocumentBySlug(slug) — 익명 서버 클라이언트로 published 1건. 기존 getDocument(owner 기반)와 분리해 책임을 명확히 한다.

8. 보안 — HTML sanitize

공개 라우트는 비로그인 노출이므로, BlockNote가 만든 HTML을 그대로 dangerouslySetInnerHTML 하기 전에 sanitize한다. sanitize-html(서버에서 실행) 또는 rehype-sanitize를 사용한다. 허용 태그: 제목/문단/목록/표/코드/인용/링크/이미지/굵게·기울임 등 BlockNote 출력 태그. script/on* 속성 제거.

왜. 본문 작성자는 인증 사용자지만, 발행물은 누구나 본다. 작성자 계정이 악성 HTML을 심으면 방문자에게 XSS가 된다. 공개 경계에서 sanitize로 차단한다.

10. 테스트

  • vitest 단위:
    • sanitize: <script>·onerror 제거, 허용 태그(table/h1/p/strong) 보존.
    • setDocumentPublished 입력 zod 검증(잘못된 입력 거부) — 로직 단위에서 가능한 범위.
  • 런타임/통합(QA): 발행 → /p/{slug} 익명 200 + 본문 노출; unpublish → /p/{slug} 404; 타인/익명이 draft 접근 불가; owner 가드(미인증 거부). 실제 Supabase로 검증(테스트 유저 정리).

11. 미해결/검토 필요

  • 공개 URL 슬러그: 현재 draft-xxxx 형태를 그대로 사용(사용자 승인). 제목 기반 예쁜 슬러그·한글 지원은 후속.
  • sanitize 라이브러리 선택(sanitize-html vs rehype-sanitize)은 구현 계획에서 확정 — 서버 실행·번들 영향 고려.
  • blocksToFullHTML이 조각 HTML을 반환하므로 공개 페이지의 래퍼/기본 스타일은 최소만(읽기 뷰 디자인은 S5).
