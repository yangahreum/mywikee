1. 배경 & 목적

별도의 LLM-Wiki(/Users/areumyang/AI/wiki, Obsidian 볼트)는 프로젝트별로 sources/(불변 원천)와 wiki/(LLM이 관리하는 마크다운)로 구성된다. 지금까지 sources/의 원본 문서는 사람이 직접 손으로 HTML/MD를 작성해 왔다.

이 프로젝트의 목적은 그 원본 문서(source 데이터)를 웹 편집기로 작성·저장하고, 저장된 데이터로부터 LLM-Wiki의 source 파일을 생성하는 도구를 만드는 것이다. 웹에 배포되면 본질적으로 사용자가 작성한 글 자체를 Supabase에 저장하는 다중 사용자 위키이며, 파일 생성은 볼트가 있는 환경에서 도는 별도 브리지다.

핵심 원칙. youllog와 동일한 순수 라이브러리 조합(Next.js + React + BlockNote + Supabase + TS, pnpm) 만으로 시작한다. 커스텀 블록·커스텀 직렬화기·AI SDK는 MVP에서 제외하고, BlockNote 기본 스키마/툴바와 내장 HTML 익스포터만 사용한다.

2. 전체 데이터 흐름

[웹 배포] 사용자 ─▶ 기본 BlockNote 에디터 ─▶ 글 저장(Supabase, 다중 사용자/RLS) ← 핵심 제품 │ [로컬/자체호스팅] source 생성 API ──────────────────┘─▶ 내장 HTML 익스포터 │ ▼ /Users/areumyang/AI/wiki/{project}/sources/{slug}.html ← 브리지

편집 상태의 진실(source of truth)은 Supabase의 documents.content(BlockNote Block[] JSON)다. HTML 파일은 그로부터 파생되는 export 산출물이며, 파일시스템 접근이 가능한 로컬/자체호스팅에서만 생성된다. 순수 웹 배포 환경에서는 export 라우트가 비활성이고 글은 DB에 그대로 보존된다.

RLS: owner_id = auth.uid() 기준으로 select/insert/update/delete 전부 제한. 각 사용자가 자기 문서만 관리하는 멀티테넌트 공유 배포. 공개 읽기/공유는 후속 슬라이스.

⚠️ slug 유니크 결정. 동일 project 내 제목 충돌 시 slug에 짧은 suffix를 붙여 충돌을 회피한다 (예: my-doc, my-doc-2). slug는 생성 시 1회 확정하고 제목 변경으로 자동 바뀌지 않는다 (export 파일 경로 안정성 위해).

에디터 동작

  • BlockNote 기본: useCreateBlockNote() + <BlockNoteView /> (mantine). 기본 블록·기본 슬래시 메뉴·기본 포매팅 툴바.
  • 제목은 BlockNote 외부의 별도 <input>으로 관리.
  • 자동저장: onChange 디바운스(예: 800ms) → documents.content(jsonb)와 title UPDATE, updated_at 갱신.

6. Source 생성 (export 브리지)

  • 서버 라우트에서 ServerBlockNoteEditor.create()의 blocksToFullHTML(content)로 HTML 문자열 생성 — 순수 라이브러리, 서버 사이드 동작.
  • 대상 경로: {WIKI_ROOT}/{project}/sources/{slug}.html
    • WIKI_ROOT = 환경변수 (기본 /Users/areumyang/AI/wiki)
    • {project}/sources/ 디렉토리가 없으면 생성
  • 환경 가드: WIKI_ROOT가 설정되어 있고 파일시스템 쓰기가 가능할 때만 export 활성. 미설정 시 라우트는 명시적 비활성 응답(글은 DB에 보존).
⚠️ 경로 안전. project/slug는 파일 경로에 직접 들어가므로 path traversal 방지를 위해 [a-z0-9-] 화이트리스트로 정규화/검증한다.

8. 테스트

  • vitest (단위): slug 유도/충돌 회피, export 경로 조합, path traversal 방어, HTML 생성 스모크.
  • Playwright (e2e): 로그인 → 새 문서 → 편집 → 자동저장 확인 → source 생성 → 파일 존재 확인 한 바퀴.

9. 미해결/검토 필요

  • 매직링크 인증 흐름 세부(youllog의 token_hash 방식 차용 여부) — 구현 계획 단계에서 확정.
  • export 시 HTML 상단에 메타(created/tags/source) 주석 포함 여부 — 기본은 순수 export, 필요 시 후속.
