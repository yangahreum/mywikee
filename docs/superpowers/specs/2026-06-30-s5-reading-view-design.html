S5 — 읽기 뷰 (청사진 Frame 2)

문서를 읽는 화면(/p/[slug])을 청사진 Frame 2와 같게 만든다. 동시에 "보기 ↔ 편집" 흐름을 바로잡는다 — 트리·목록·카드에서 문서를 클릭하면 읽기 뷰로 가고, 읽기 뷰의 편집 버튼으로 편집기에 들어간다. 신규 문서만 바로 편집기로 연다.

청사진 Frame 2/p/[slug]blocksToHtml→sanitizeTOC 스크롤스파이보기/편집 분리

1. 배경 & 동기

현재 트리·Recently edited·Jump back in에서 문서를 클릭하면 바로 편집기(/edit/[id])로 간다. 이는 "클릭=보기"라는 일반적 멘탈 모델과 어긋난다(사용자 지적). 청사진도 읽기(Frame 2)와 편집(Frame 3)이 분리돼 있다. S5는 읽기 뷰를 만들고 흐름을 정상화한다.

흐름 (S5 후). 문서 클릭(트리/목록/카드) → /p/[slug] 읽기 뷰. 읽기 뷰의 Edit 버튼(owner) → /edit/[id]. New Page/New Entry → /edit(빈 문서라 바로 편집).
Publish(공개)와의 관계. S5는 owner의 내부 읽기 뷰를 만든다(자기 문서를 RLS로 조회). 익명 공개 읽기(비로그인 노출)는 Publish 기능(status='published' + 공개 RLS)이 들어가는 S4 이후 같은 /p/[slug] 라우트에서 활성화된다. S5에선 sanitize·읽기 UI를 완성해 두고, S4에서 RLS만 확장하면 공개로 전환된다.

2. 보기/편집 흐름 변경 (클릭 라우팅)

위치현재S5 후
트리 문서 행 (TreeNodeRow)/edit/{id}/p/{slug}
Recently edited 행 (RecentlyEditedTable)/edit/{id}/p/{slug}
Jump back in 카드 (JumpBackIn)/edit/{id}/p/{slug}
New Page / New Entry/edit/{id} (빈 문서)그대로 (빈 문서 → 바로 편집)
읽기 뷰 Edit 버튼 (신규)—/edit/{id}

이를 위해 slug를 클릭 경로까지 전파한다 — DashboardDoc·DocRow(트리)에 slug 필드를 추가하고, 각 조회(select)에 slug를 포함한다.

3. 라우트 /p/[slug] (서버 컴포넌트)

  • 파일: app/p/[slug]/page.tsx. getUser 가드(S5는 내부 읽기 — 미인증 /login; 공개는 S4 후).
  • getDocumentBySlug(slug, ownerId) 조회 → 없으면 notFound().
  • 공통 AppShell(사이드바 트리 + 토픽바) 안에 읽기 콘텐츠. 트리도 함께 로드(getTree).
  • 토픽바 우측에 Edit 버튼(owner이면) → /edit/[id]. (Publish 버튼은 S4.)

조회 헬퍼

lib/documents/get-by-slug.ts: getDocumentBySlug(slug, ownerId) — slug + owner로 1건(id,title,slug,content,project,folder_id,updated_at). owner 가드(자기 문서). 공개 조회는 S4에서 분리 헬퍼.

4. 본문 렌더 (blocksToHtml → sanitize)

  • blocksToHtml(content)(기존, 서버) → HTML 문자열.
  • sanitize: sanitize-html 도입(서버 실행). 허용 태그(제목/문단/목록/표/코드/인용/링크/이미지/강조 등 BlockNote 출력) + script/on* 제거. 래퍼 lib/render/sanitize.ts: sanitizeHtml(html).
  • 렌더: <div dangerouslySetInnerHTML={{__html: safe}}>에 읽기 타이포 스타일(청사진 본문: serif 15/1.65 등) 적용 — 본문 컨테이너에 prose 류 클래스(직접 정의).
  • 왜 sanitize: 지금은 owner 본인 글이라 위험이 낮지만, S4 공개 전환 시 비로그인 노출되므로 경계에서 미리 정제한다(Publish 스펙 §8).

5. TOC (On this page) — 헤딩 추출 + 스크롤 스파이

  • 본문 HTML에서 h2/h3 헤딩 추출 → 목차. 각 헤딩에 id 부여(slugify), TOC 링크 #id.
  • 추출 순수 함수 lib/render/toc.ts: extractToc(html) → {id, text, level}[] + id가 주입된 html 반환(또는 별도) — TDD.
  • 스크롤 스파이(현재 섹션 활성): 클라이언트 IntersectionObserver 컴포넌트 TocNav. 활성 항목 청사진처럼 좌측 2px primary 바 + text-ink/600.
  • Related pages: 같은 폴더 또는 같은 project 문서 N건(간단). 데이터 없으면 섹션 숨김 또는 placeholder.

7. Reading time

lib/render/reading-time.ts: readingTime(text) → 분(올림). BlockNote content에서 텍스트 추출(또는 sanitize된 html에서 텍스트) → 단어/문자 수 → 분당 추정(한글 고려: 분당 ~500자 등). 순수 함수 TDD.

9. 테스트

  • vitest 단위:
    • sanitize: <script>·onerror 제거, 허용 태그(h2/p/ul/pre/strong/a/img) 보존.
    • toc: h2/h3 추출, id slugify(중복 처리), 헤딩 없으면 빈 목차.
    • reading-time: 빈/짧은/긴 텍스트 분 계산, 최소 1분.
    • Article: 제목·브레드크럼·본문(sanitized) 렌더, Edit 버튼(editHref 있을 때).
    • TocNav: 목차 항목 렌더(스크롤 스파이는 부분 — 활성 표시 로직).
    • 회귀: slug 추가가 기존 DashboardDoc/DocRow 테스트와 정합(매핑 갱신).
  • QA/런타임: 트리/목록/카드 클릭 → 읽기 뷰; Edit 버튼 → 편집기; 신규 → 편집기; TOC 스크롤 스파이; sanitize(악성 HTML 차단). 실제 Supabase.

10. 미해결 / 검토

  • 본문 타이포 적용 방식: BlockNote가 만든 HTML에 청사진 읽기 스타일을 입히는 법 — 전역 prose 클래스(자식 셀렉터) vs Tailwind typography 플러그인. 계획에서 확정(의존성 최소 위해 직접 CSS 권장).
  • blocksToFullHTML 래퍼: 완전 HTML(html/body 포함)일 수 있어 본문 조각만 추출 필요할 수 있음 — 계획에서 출력 형태 확인 후 처리.
  • slug 없는 기존 문서: 모든 문서가 slug UNIQUE NOT NULL이라 존재(0002). draft-xxxx 형태 그대로 URL.
  • Related pages: 같은 폴더 기준 vs project 기준 — 데이터 적으면 placeholder. 계획에서 단순화.
  • 확인 모달(삭제) 커스텀화는 이 슬라이스와 별개(후속).
