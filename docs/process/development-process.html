wiki 개발 프로세스

이 프로젝트가 한 기능(슬라이스)을 만드는 방식. 설계 → 계획 → 팀 구현 → 검증 → dev 확인의 생성-검증 루프다. CLAUDE.md가 규칙(정책)이라면, 이 문서는 그 규칙이 실제로 어떻게 돌아가는지 — 흐름·역할·산출물을 한눈에 본다.

1. 전체 흐름

요구 / 청사진 │ ▼ ┌───────────┐ 리뷰 게이트 │ ① 설계(스펙) │ ───────────► 사용자 승인 └───────────┘ │ ▼ ┌───────────┐ │ ② 계획(plan)│ TDD Task 분할 └───────────┘ │ ▼ ┌──────────────────────────────────┐ │ ③ 구현 루프 (에이전트 팀) │ │ implementer ─► reviewer ─► qa │ ◄─ CHANGES_REQUESTED 시 수정 후 재확인 │ (TDD) (정적) (실행) │ └──────────────────────────────────┘ │ 체크포인트 보고 ▼ ┌───────────┐ │ ④ 검증 │ tsc + test + 회귀 + 기능표 대조 └───────────┘ │ ▼ ┌───────────┐ 피드백 루프 │ ⑤ dev 확인 │ ───────────► 폴리시·버그픽스(메인 직접) / 신규 기능은 ①로 └───────────┘

2. 단계별

① 설계 — 스펙 (메인 작성)

  • brainstorming으로 요구·범위를 정렬한다. 작게 시작하고, 미루는 항목은 후속 슬라이스로 명시한다.
  • HTML 스펙을 docs/superpowers/specs/에 작성한다. 최상단에 기능정의 표와 그에 매칭되는 작업 표를 두고(CLAUDE.md §2), 본문에 데이터 모델·UI 픽셀·테스트 범위를 정리한다.
  • UI는 청사진(Digital Sanctuary)과 최대한 동일하게 — 색·폰트·간격·radius를 그대로 옮긴다.
  • 커밋하고 풀패스로 공유 → 사용자 리뷰 게이트. 승인 후에만 다음으로 간다.

② 계획 — plan (메인 작성)

  • 스펙을 docs/superpowers/plans/의 TDD Task 분할 계획서로 옮긴다.
  • 각 Task = 실패 테스트 → 최소 구현 → 통과 → 커밋 단위. 작업 표 항목과 Task를 매핑한다.
  • 각 Task에 정확한 파일 경로·코드·검증 명령·커밋 메시지를 적는다(zero-context 엔지니어 가정).

③ 구현 루프 — 에이전트 팀 (nextjs-build-orchestrator 조율)

  • 연관 Task를 묶어 implementer에 위임한다 — TDD, 청사진 픽셀 충실, 계획 명시 파일만 커밋.
  • 묶음 단위로 code-reviewer 정적 리뷰 — 계획 준수·보안(getUser/RLS/zod)·타입 일관성·테스트 품질·기능표 대조(스펙의 기능/작업이 누락 없이 구현됐는지).
  • qa-verifier 실행 검증 — tsc/test/build + 경계면 교차(조회→소비 shape) + 회귀.
  • 리뷰가 CHANGES_REQUESTED면 implementer가 지적만 수정 → 재확인. 깨끗이 끝나면 체크포인트 1줄 보고 후 다음.
왜 셋을 분리하나. 구현자·검토자·검증자를 나누면 "자기 코드를 자기가 통과시키는" 편향이 차단된다. 정적 결함(리뷰)과 실행 결함(QA)은 다른 시각에서 잡힌다.

④ 검증 원칙

  • 매 슬라이스 pnpm tsc --noEmit + pnpm test 통과(완료 게이트).
  • 회귀 동반: 이전 테스트를 함께 돌려 영향 확인. 신규 구현이 기존 테스트를 깨면 테스트가 아니라 신규 코드를 수정한다. 단 요구사항(스펙)이 바뀐 경우의 테스트 갱신은 "의도된 변경"으로 구분한다.
  • 기능표 대조: 스펙 최상단 기능정의 표·작업목록과 구현을 대조해 누락 없는지 확인(CLAUDE.md §3).
  • 환경 의존 분리: 실제 DB/인증 왕복은 환경(마이그레이션·세션) 의존이라, 미적용 시 "미검증"으로 분리 표기하고 통과로 적지 않는다.

⑤ dev 확인 — 사용자

  • 사용자가 pnpm dev로 직접 확인하고 피드백한다.
  • 작은 시각 폴리시·버그픽스(예: 배경색, 커서, 누락 요소)는 메인이 직접 수정·검증·커밋한다.
  • 새 기능 요구는 다시 ①설계부터.
  • 외부 DB 스키마 변경은 사용자가 수동 적용(마이그레이션 SQL 안내).

3. 역할

주체책임안 하는 것
메인 (리더)의도·설계(스펙)·계획·팀 조율·리뷰 종합·직접 폴리시·메모리(필요 시 직접 구현하나 기능 구현은 위임)
nextjs-implementer단일/묶음 Task를 TDD로 구현·커밋리뷰/판정
code-reviewerdiff 정적 리뷰·PASS/CHANGES_REQUESTED 판정코드 수정·실행
qa-verifier실행 검증(tsc/test/build)·회귀·경계면코드 수정·커밋

모든 에이전트는 model: opus. 에이전트·스킬 정의는 .claude/agents/, .claude/skills/. 이 팀은 harness 메타 스킬로 생성되었고, 다른 스택이 필요하면 범용화하지 않고 harness로 그 프로젝트용 팀을 새로 만든다.

4. 공통 규약

  • git: master 직접 커밋, 명시 파일만 git add(공유 모노레포 — git add -A 금지). 커밋 끝에 Co-Authored-By.
  • 언어: 모든 답변·문서는 한글.
  • 문서 공유: 스펙·계획은 풀 로컬 패스로 링크.
  • 메모리: 결정·피드백을 ~/.claude/.../memory/에 기록(다음 세션 연속성).
  • 중간 산출물: 크면 _workspace/에 보존(감사 추적). 최종은 계획 명시 파일만 커밋.

6. 한 슬라이스의 생애 (예: S5 읽기 뷰)

사용자: "트리에서 파일 클릭하면 바로 편집되는 게 이상한데" → 메인: 보기/편집 흐름 문제 진단 → S5 읽기뷰가 해법 → ① 스펙 작성(청사진 Frame 2 + 흐름) → 커밋 → 공유 → ② 계획서(9 Task: sanitize/toc/Article/TocNav/라우트/라우팅 변경) → ③ implementer 묶음(1~4 lib → 5~7 컴포넌트 → 8~9 통합) 각 묶음 후 reviewer, 마지막 qa → ④ tsc/test(76)/build 통과, 경계면 정합, 기능표 대조 → ⑤ 사용자 dev 확인 → 폴리시/다음 슬라이스

이 문서는 프로세스가 진화하면 갱신한다. 규칙의 단일 출처는 CLAUDE.md.
